# ADR-002: First production deployment on x86 with Debian 12

## Goal
Validate the first production deployment of Radar on a single cloud VM, using a stable and operationally simple setup that supports `radar-core`, PostgreSQL, and Metabase, while keeping the door open for a later ARM evaluation once the x86 production path is proven.

## Status
Accepted

## Context
Radar includes `radar-core`, PostgreSQL, and Metabase. `radar-core` processes large financial datasets in Python 3.13 with Polars, NumPy, Numba, and multiprocessing, and it will be executed on a scheduled basis through `systemd timer + service`. The current priority is to validate a stable first production deployment, learn cloud deployment end to end, and establish the operating model before exploring ARM and further optimizations. [man7](https://man7.org/linux/man-pages/man5/systemd.timer.5.html)

## Decision
The first production version will run on an **x86_64** VM with **Debian 12 (Bookworm)** as the host operating system. `radar-core`, PostgreSQL, and Metabase will run on that host using a simple operational model, with `radar-core` started by `systemd timer` and `systemd service` so that it only consumes resources during its scheduled runs. [siberoloji](https://www.siberoloji.com/how-to-run-docker-containers-on-debian-12/)

## Options considered

#### 1. ARM64 (Hetzner CAX / Scaleway ARM)  
Benefit: potentially better price/performance and a strong candidate for future optimization.  
Risk: additional validation is needed for build and runtime compatibility with native dependencies, and the stack must be explicitly proven on ARM. [adr.github](https://adr.github.io)

#### 2. x86_64 (Hetzner CX / OVH VPS)  
   Benefit: lower operational uncertainty, natural continuity with the current development environment, and a straightforward path to validate the existing stack.  
   Risk: it may not be the most efficient long-term option if ARM later proves to perform better. [ovhcloud](https://www.ovhcloud.com/en/vps/vps-belgium/)

#### 3. Debian vs. Ubuntu  
   Benefit: the same stack can be used for both x86 and ARM, and coherent host base for a `bookworm`-based container image like `radar-core`.
   Risk: the choice of Debian over Ubuntu may limit the availability of certain packages or community support, which could impact future development and maintenance. [shape](https://shape.host/resources/how-to-install-docker-on-debian-12)

## Consequences
The first phase is optimized for **stability and operational learning**, not for absolute minimum cost. This reduces the risk of dependency issues, keeps the relationship between host and container simple, and makes `systemd` a clean execution mechanism for scheduled runs. [documentation.suse](https://documentation.suse.com/smart/systems-management/html/systemd-working-with-timers/index.html)
As a consequence, ARM is deferred to a later phase of work: first Radar will be validated in production on x86, and then separate tasks can be created for “build for ARM,” “test on ARM,” and eventually “deploy on ARM”. [github](https://github.com/joelparkerhenderson/architecture-decision-record)

## Main reasons
- The immediate goal is to reach production and learn the full cloud deployment lifecycle.
- The current stack already works locally on x86 and under Docker Desktop with WSL2, which reduces initial surprises.
- Debian 12 is a stable, lightweight, and coherent host base for a `bookworm`-based container image. [shape](https://shape.host/resources/how-to-install-docker-on-debian-12)
- `systemd timer + service` fits well with a single VM and periodic batch workloads. [man7](https://man7.org/linux/man-pages/man5/systemd.timer.5.html)

## Expected result
The expected outcome is an operational first version of Radar in the cloud with:
- `radar-core` running only in its scheduled windows.
- PostgreSQL and Metabase continuously available.
- A clear path to automate deployments with GitHub Actions.
- A technical baseline that will later allow a data-driven comparison between x86 and ARM. [aws.amazon](https://aws.amazon.com/blogs/architecture/master-architecture-decision-records-adrs-best-practices-for-effective-decision-making/)
